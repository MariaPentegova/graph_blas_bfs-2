# Сравнение BFS: классическая реализация (CSR) и GraphBLAS

## О проекте

В данном проекте реализованы и протестированы два варианта алгоритма обхода графа в ширину (BFS):

* **Классический BFS**: на основе очереди и CSR-представления графа.
* **BFS на основе линейной алгебры**: через библиотеку **SuiteSparse:GraphBLAS**.

## Реализованные операции

##  BFS с построением дерева обхода (parent BFS)

### Классическая версия
Обход в ширину с использованием стандартной очереди.

```c
BFSResult* classic_bfs(int n, int* row_ptr, int* col_idx, int source);
```

**Возвращает структуру `BFSResult` со следующими полями:**
* `parent` — массив родителей (для каждого узла — его предок в BFS-дереве)
* `level` — массив уровней (расстояние от источника)
* `visited_count` — количество посещённых узлов

### Версия GraphBLAS
Итеративное умножение вектора фронта на матрицу смежности графа.

```c
int* graphblas_bfs(GraphBLASGraph* graph, int source, double* time_ms, int** level);
```

* **Возвращает:** массив `parent` (родителей).
* **Выходные указатели:** через `level` возвращается массив уровней, через `time_ms` — время выполнения алгоритма.

##  Multi‑source BFS (обход из нескольких источников)

### Классическая версия
Использует обычную очередь, но изначально в неё добавляются все заданные источники.

```c
void classic_bfs_multisource(int n, int* row_ptr, int* col_idx, int* sources, int num_sources, BFSResult* result);
```

* **Принцип работы:** обход распространяется одновременно от всех источников. Каждая вершина получает родителя от того источника, который достиг её первым.
* **Особенности:** корректно работает на несвязных графах (каждый источник обходит свою компоненту связности).
* **Сложность:** Память — $O(V + E)$, Время — $O(V + E)$.

### Версия GraphBLAS
Вектор фронта инициализируется всеми источниками одновременно.

```c
int* graphblas_bfs_multisource(GraphBLASGraph* graph, int* sources, int num_sources, double* time_ms, int** level);
```

* **Принцип работы:** на каждой итерации выполняется умножение вектора фронта на матрицу смежности. Все новые вершины обрабатываются параллельно за одну матричную операцию.
* **Особенности:** на больших графах может быть эффективнее классической версии за счёт встроенной параллелизации.
* **Сложность:** Память — $O(V + E)$ + накладные расходы библиотеки GraphBLAS.

## Вспомогательные функции

* `load_matrix_market` — загрузка графа из файла формата Matrix Market (`.mtx`).
* `timer_start` / `timer_elapsed_us` — высокоточный таймер на основе функции `gettimeofday()` (измерение в микросекундах).
* `save_bfs_result` — сохранение полученных результатов работы BFS в файл.

## Сборка и запуск

### Требования
Для установки необходимых зависимостей в Ubuntu/Debian выполните:

```bash
sudo apt install libsuitesparse-dev cmake build-essential
```

### Сборка
Компиляция проекта выполняется стандартным способом через CMake:

```bash
mkdir build && cd build
cmake ..
make
```

### Запуск тестов
Для проверки корректности работы всех реализаций запустите:

```bash
ctest
```

Также можно запустить отдельные исполняемые файлы тестов:

```bash
./test_classic_bfs
./test_graphblas_bfs
```

### Запуск бенчмарка
Для проведения анализа производительности используйте следующую команду:

```bash
./bfs_analysis ../graphs/название_графа.mtx источник
```

##  Анализ результатов

1. **delaunay_n16 (структурированный граф) — победа GraphBLAS**
   * **Результат:** GraphBLAS быстрее в **2.75 раза** (в среднем `178 мкс` против `490 мкс`).
   * **Диапазон:** Минимальное ускорение — `1.68x`, максимальное — `3.92x`.
   * **Причина:** Равномерная структура и широкие фронты BFS эффективно параллелизуются и обрабатываются матричными операциями GraphBLAS.

2. **com-Amazon (реальный граф) — ничья**
   * **Результат:** Разница около **7%** в пользу Classic (`2 900 мкс` против `3 100 мкс`).
   * **Диапазон:** В одном из запусков GraphBLAS показал ускорение `1.32x`, в остальных Classic был незначительно быстрее или шел вровень.
   * **Причина:** Выраженная структура сообществ внутри графа не дает явного математического преимущества ни одному из подходов.

3. **kron_g500-logn16 (безмасштабный граф) — победа классики**
   * **Результат:** Classic BFS быстрее на **28%** (`3 350 мкс` против `4 300 мкс`).
   * **Диапазон:** Ускорение классической версии варьируется от `1.15x` до `1.40x`.
   * **Причина:** Очень маленький диаметр графа (всего 6 уровней). Стандартная очередь работает эффективнее на таких "мелких" BFS-деревьях, исключая накладные расходы на матричную математику.

## Выводы

* **Зависимость от топологии:** производительность алгоритмов напрямую определяется топологией исследуемого графа, а не только его физическим размером.
* **Эффективность GraphBLAS:** подход максимально эффективен на графах с равномерной структурой и широкими фронтами (`delaunay_n16`) — зафиксировано ускорение до **2.75x**.
* **Эффективность классического BFS:** стандартный алгоритм показывает преимущество до **28%** на графах с малым диаметром (`kron_g500-logn16`).
* **Поведение на реальных данных:** на реальных графах (`com-Amazon`) оба подхода демонстрируют сопоставимые результаты с незначительной разницей в **~7%**.
* **Корректность реализаций:** все тесты успешно проходят (**100% passed**), оба алгоритма дают идентичные результаты по достижимости/недостижимости узлов (`Results consistent: YES`).
* **Потенциал оптимизации:** сильные стороны GraphBLAS полностью раскрываются на плотных графах с равномерной структурой, где параллельные матричные операции компенсируют накладные расходы.
* **Стабильность измерений:** серия из 10 запусков показала низкую вариативность данных — стандартное отклонение времени выполнения не превышает **15%**.
